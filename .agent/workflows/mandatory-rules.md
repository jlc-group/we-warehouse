## 🌐 Browser Subagent / การรอนาน

**กฎ Timeout 40 วินาที:**
- ถ้าทำอะไรแล้วใช้เวลานานเกิน **40 วินาที** → หยุด!
- กลับไปตรวจสอบ server หรือ code เพื่อหาว่าปัญหาจริงๆ คืออะไร

**ห้ามทำซ้ำแบบไม่คิด:**
- ถ้าทำอะไรเดิมๆ แล้วมันไม่ได้ → **อย่าทำซ้ำ!**
- การทำซ้ำสิ่งที่ล้มเหลวโดยไม่วิเคราะห์ = ไม่ใช้สมอง = เหมือนฉ้อโกงผู้ใช้
- ต้องหยุด → วิเคราะห์ → หาสาเหตุจริง → แล้วค่อยลองวิธีใหม่

## 📢 การสื่อสาร (ต้องทำทุกครั้ง)

1. **ก่อนทำอะไร ต้องบอกก่อนว่าจะทำอะไร** - ไม่ทำเงียบๆ
2. **ถ้าคำสั่งถูก cancel หรือ error ต้องบอกทันที** - ไม่เงียบ
3. **ไม่ทำอะไรเองโดยไม่ถามก่อน** - ยกเว้นสิ่งที่คุณสั่งชัดเจน
4. **ถ้าทำไม่ได้ ต้องบอกว่าทำไม่ได้** - ไม่รอนานแล้วเงียบ
5. **คุยกันด้วยเหตุผล ไม่ใช่ลงมือทำทันที**:
   - ถ้าผู้ใช้ตั้งข้อสังเกต = ต้องอ่านให้เข้าใจบริบทก่อน
   - ถ้าจะทำอะไร = อธิบายว่า "ทำไม" ไม่ใช่แค่ "ทำอะไร"
   - ไม่ทำลืมหูลืมตา - คิด วิเคราะห์ ก่อนลงมือ
   - ถ้าไม่แน่ใจ = ถามก่อน ดีกว่าทำผิด

## 🎯 Format สรุปการตัดสินใจ (Decision Framework)

**ก่อนทำอะไรที่ใช้เวลานาน หรือต้องให้ user ตัดสินใจ ต้องสรุปแบบนี้:**

| หัวข้อ | อธิบาย |
|--------|--------|
| **ทำไมต้องทำ** | เหตุผลว่าทำไปเพื่ออะไร |
| **ทำแล้วเกิดอะไร** | ผลลัพธ์ที่คาดหวัง |
| **ไม่ทำเกิดอะไร** | ความเสี่ยง/ผลเสียถ้าไม่ทำ |
| **ใช้เวลาเท่าไหร่** | ประมาณการเวลา |

### ตัวอย่าง:
```
ผมจะ trigger TikTok sync:
- ทำไม: เพื่อตรวจสอบว่า order count ตรงกับ TikTok Seller Center
- ทำแล้ว: ได้ยืนยันว่า sync ทำงานถูกต้อง + orders ครบ
- ไม่ทำ: ไม่รู้ว่า sync มีปัญหาหรือเปล่า / orders อาจหายไป
- เวลา: ประมาณ 5-10 นาที
```

**กฎ:**
- ❌ ห้ามรัน task หนักโดยไม่บอกก่อน
- ❌ ห้ามรอนานโดยไม่อธิบายว่ารออะไร
- ✅ ต้องให้ user ตัดสินใจได้อย่างมีข้อมูล

## ⛔ ห้ามทำเด็ดขาด

1. **ห้าม git revert/reset โดยไม่ถาม** - ต้องถามและบอกว่าจะทำอะไร
2. **ห้ามลบไฟล์โดยไม่ถาม** - ต้องลิสต์ไฟล์และขออนุมัติ
3. **ห้ามแก้ไข .env โดยไม่ถาม** - config สำคัญมาก
4. **ห้ามสมมติ** - ต้องเช็คข้อมูลจริงก่อนพูด
5. **ห้ามทำหลายอย่างพร้อมกัน** - ทำทีละขั้น verify ก่อนไปต่อ
6. **ห้ามแก้ไข Deployment Config** - ไฟล์ config ที่ใช้ deploy ต้องไม่ถูกเปลี่ยน:
   - `nginx.conf` - Nginx configuration
   - `ecosystem.config.cjs` - PM2 configuration  
   - `app/core/config.py` - Port settings (APP_PORT: 9203)
   - `frontend/vite.config.ts` - Proxy settings (target: localhost:9203)
   - `start.sh`, `start_backend.sh`, `stop.sh`, `status.sh` - Shell scripts
   - `weorder.service` - Systemd service unit
   - `scheduler.py` - Background sync scheduler
   - ถ้าจำเป็นต้องแก้ ต้องถามก่อนเสมอ!
7. **ห้ามโกหก/หลอกลวง** - ต้องพูดความจริงเท่านั้น:
   - ถ้าไม่รู้ ต้องบอกว่า "ไม่รู้"
   - ถ้าทำไม่ได้ ต้องบอกว่า "ทำไม่ได้"
   - ห้ามพูดส่งเดชแล้วค่อยมาแก้ทีหลัง
   - ห้ามโยนความผิดให้คนอื่นหรือสิ่งอื่น
   - ถ้าผมผิด ผมต้องยอมรับทันที
8. **ห้ามพูดก่อนตรวจสอบ** - ต้องเช็คข้อมูลจริงก่อนพูด:
   - ห้ามสมมติหรือเดาก่อนตรวจสอบ
   - ต้องใช้ tool ตรวจสอบก่อน (run_command, view_file, etc.)
   - การพูดส่งเดชโดยไม่มีหลักฐาน = อวดรู้ = ไม่ยอมรับ
   - ถ้าไม่แน่ใจ ตรวจสอบก่อน ไม่ใช่พูดก่อน
9. **ห้าม Port ซ้ำซ้อน** - ก่อนแนะนำ/รัน service:
   - ต้องเช็ค `netstat -ano | findstr ":<PORT>"` ก่อน
   - ถ้า port ถูกใช้งานอยู่ → ห้ามใช้ซ้ำ
   - ต้องตรวจสอบว่า port ไหนว่างก่อนแนะนำ
   - weorder: 9203 (backend), nova-ai: 8765 (backend)

## 🔄 Sync Architecture (สำคัญ!)

### หลักการ:
| Component | หน้าที่ | ทำไม |
|-----------|--------|------|
| **Webhooks** | Primary - Real-time updates | ได้ orders ทันทีที่มีการเปลี่ยนแปลง |
| **Sync** | Backup - Safety net | จับ orders ที่ webhook พลาด (server restart, network issue) |

### ตั้งค่าที่ถูกต้อง:
- **Sync Frequency**: วันละ 2 ครั้ง (08:00, 20:00) - ไม่ใช่ทุกชั่วโมง
- **TikTok Lookback**: 1 วัน (optimized จาก 3 วัน เพื่อความเร็ว)
- **TikTok Status Filters**: AWAITING_SHIPMENT, ON_HOLD เท่านั้น (ไม่ต้องดึงทุก status)
- **เหตุผล**: Sync บ่อยเกินไป = ข้อมูลเด้ง, ดึงมากเกินไป = ช้า 28 นาที

### ห้ามทำ:
- ❌ ตั้ง Sync ทุกชั่วโมง (ทำให้ UI เด้ง)
- ❌ ดึง 30 วันย้อนหลัง (ช้ามาก)
- ❌ ดึงทุก status (ไม่จำเป็น, Webhooks handle real-time)
- ❌ ปิด Sync ทั้งหมด (Webhooks ไม่ reliable 100%)
